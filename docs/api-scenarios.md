# 🛡️ API 보안 방어 시나리오 (API Scenarios)

단순한 API 스펙 나열이 아닌, **실제 해킹 위협(Threat) 모델링**을 기반으로 `security-auth-core`가 어떻게 방어하는지 보여주는 스토리텔링 시나리오입니다.

모든 시나리오는 IDE(IntelliJ, VSCode)에서 `examples/scenarios.http` 파일을 통해 클릭 한 번으로 직접 실행해 볼 수 있습니다.

---

## 🎭 시나리오 1: RTR(Refresh Token Rotation) 기반 토큰 탈취 방어

### [상황]
해커가 PC방이나 공용 네트워크에서 네트워크 스니핑 혹은 XSS 공격을 통해 사용자의 `RefreshToken`을 탈취했습니다.

### [방어 흐름]
1. **정상 유저의 토큰 갱신**: 사용자가 앱을 사용하며 정상적으로 탈취당했던 기존의 `RefreshToken`을 사용해 새로운 토큰 세트를 발급받습니다.
2. **이전 토큰 무효화**: 서버의 `RefreshTokenRepository`는 즉시 해당 이전 토큰을 폐기(Delete)하고 새로운 토큰 세트를 Redis 캐시에 저장합니다.
3. **해커의 접근 시도**: 해커가 자신이 탈취했던 기존 `RefreshToken`으로 서버에 토큰 갱신 API(`/api/v1/auth/refresh`)를 몰래 호출합니다.
4. **방어 성공 (401 Unauthorized)**: 서버는 이미 폐기되어 Redis에 존재하지 않거나 일치하지 않는 토큰임을 즉시 감지하고 접근을 거부합니다. 나아가 "탈취된 토큰의 재사용"으로 의심되는 경우, 해당 유저의 **모든 세션(토큰)을 블랙리스트 처리**하여 계정을 완전 격리시킬 수 있습니다.

> 💡 **검증 방법**: `examples/scenarios.http`에서 **[3. 토큰 갱신]**을 2번 연속으로 실행해보세요. 두 번째 요청은 이미 로테이션된 토큰을 사용하므로 반드시 401 에러가 발생하여 보안 로직이 작동함을 증명합니다.

## 🎭 시나리오 2: 하이브리드 암호화 (E2EE) 중간자 공격 방어

### [상황]
Man-in-the-Middle (MitM) 공격자가 통신망을 장악하여 패킷을 가로채고 있습니다. (HTTPS가 프록시 등에 의해 무력화된 최악의 상황을 가정)

### [방어 흐름]
1. 클라이언트는 로그인 비밀번호나 결제 정보 등 민감한 데이터를 임의로 생성한 **대칭키(AES)**로 암호화합니다.
2. 그리고 그 AES 키를 다시 서버가 배포한 **공개키(RSA)**로 한 번 더 암호화하여 서버로 전송합니다.
3. 공격자는 네트워크 패킷에서 `encryptedData`와 `encryptedAesKey`를 온전히 탈취하는 데 성공합니다.
4. **방어 성공**: 하지만 서버 안에 깊숙이 보관된 **개인키(Private Key)**가 없으므로 공격자는 AES 키를 복호화할 수 없고, 결국 본문 데이터(비밀번호)에도 영원히 접근할 수 없습니다.

> 💡 **검증 방법**: `examples/scenarios.http`의 **[4. 하이브리드 암호화 테스트]**를 실행해 보세요. 원본 JSON 파일(`encrypt-request.json`)을 뜯어보시면 실제 평문은 네트워크 상의 그 어디에도 노출되지 않음을 확인할 수 있습니다.
