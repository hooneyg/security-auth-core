# ADR-002: 하이브리드 암호화 전략 선택 (RSA + AES-GCM)

## 📌 상태 (Status)
**Accepted**

## 📝 컨텍스트 (Context)
클라이언트(웹/모바일)와 서버 간, 또는 내부 시스템 간에 민감한 개인정보(PII: 주민등록번호, 결제 정보, 비밀번호 등)를 평문으로 전송하는 것은 HTTPS 환경이라 하더라도 엔드포인트 탈취나 프록시 스니핑 등의 위험(E2EE 부재)이 존재합니다. 애플리케이션 레벨의 페이로드 암호화가 필수적인 상황에서, 단일 암호화 알고리즘만 사용할 경우 **보안성과 처리 성능 간의 트레이드오프**가 발생합니다.

## 🎯 결정 (Decision)
- **결정**: 비대칭키(`RSA-2048`)와 대칭키(`AES-256-GCM`)를 결합한 **하이브리드 암호화 아키텍처(Hybrid Encryption)**를 채택합니다.
- **방식**: 실제 민감한 대용량 페이로드는 빠르고 안전한 `AES-256`으로 암호화하고, 이때 사용된 '일회용 AES 대칭키'만을 서버의 `RSA` 공개키로 암호화하여 함께 전송합니다.

## 🔍 결정 근거 (Rationale)

### 대칭키 분배 문제 (Key Distribution Problem)의 근본적 해결
AES와 같은 대칭키 암호화는 속도가 매우 빠르지만, 클라이언트와 서버가 **"어떻게 똑같은 키를 안전하게 나누어 가질 것인가?"**라는 치명적인 문제가 있습니다.

```mermaid
flowchart TD
    subgraph "대칭키 단독 사용 시의 딜레마"
        Client["클라이언트"] -- "데이터 + AES키 전송" --> Hacker{{"해커 (네트워크 스니핑)"}}
        Hacker -- "AES키 획득 완료!" --> Server["서버"]
        style Hacker fill:#DC2626,color:#fff
    end
```

이 문제를 해결하기 위해 RSA를 도입하여, **서버만이 풀 수 있는 자물쇠(공개키)**로 AES 키를 안전하게 감싸서 전달하는 방식을 택했습니다.

### 각 알고리즘의 단점 상호 보완
- **RSA의 한계 극복**: RSA는 연산이 매우 무겁고, 무엇보다 키 길이(2048bit)보다 긴 데이터를 암호화할 수 없다는 치명적 단점(Payload Size Limit)이 있습니다. 하이브리드 방식은 RSA로 짧은 길이의 AES 키만 암호화하므로 이 문제를 완벽히 회피합니다.
- **AES의 한계 극복**: 매 요청마다 새로운 일회용 AES 키와 IV(Initialization Vector)를 난수로 생성(Randomization)하여 전송하므로, AES 키가 유출되더라도 과거의 데이터는 안전한 **전방향 안전성(Perfect Forward Secrecy)**을 확보했습니다.

## ⚖️ 대안 평가 (Alternatives)

| 대안 | 장점 | 단점 | 탈락 사유 |
|:---|:---|:---|:---|
| **AES-256 단독** | - 암복호화 속도가 매우 빠름<br>- 대용량 데이터 처리에 적합 | - 클라이언트 코드에 대칭키를 하드코딩해야 함 (심각한 보안 취약점) | 근본적인 키 탈취 방어 불가 |
| **RSA-2048 단독** | - 키 분배가 매우 안전함 (공개키 노출 무관) | - 연산 속도가 AES 대비 수백 배 느림<br>- **긴 문자열(JSON 전체) 암호화 불가** | 대용량 JSON 페이로드 처리 불가능 |

## 🚀 결과 및 영향 (Consequences)
1. **아키텍처 복잡도 증가**: 클라이언트 측(JS/Swift 등)에서도 단순 암호화가 아닌, 난수 키 생성 -> AES 암호화 -> RSA 암호화라는 3단계 프로세스를 구현해야 하는 프론트엔드 종속성이 생겼습니다.
2. **최고 수준의 보안성 획득**: 레인보우 테이블 공격(Rainbow Table Attack), 패딩 오라클 공격(Padding Oracle Attack)에 대해 내성을 갖춘 가장 안전한 형태의 E2EE(End-to-End Encryption) 통신 채널이 확립되었습니다.
