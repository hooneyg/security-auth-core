# ADR-001: JWT 서명 알고리즘 선택 (HS256 vs HS512)

## 📌 상태 (Status)
**Accepted**

## 📝 컨텍스트 (Context)
보안 인증 코어(`security-auth-core`)에서 발급하는 JWT(JSON Web Token)의 무결성을 보장하기 위해 서명(Signature) 알고리즘을 선택해야 합니다. 일반적인 토이 프로젝트나 레퍼런스 구현에서는 관례적으로 `HS256(HMAC SHA-256)`을 사용하지만, 본 프로젝트는 **엔터프라이즈 레벨의 보안성**을 목표로 하므로 보다 강력하고 미래 지향적인 기준이 필요했습니다.

## 🎯 결정 (Decision)
- **결정**: `HS512` (HMAC SHA-512) 대칭키 암호화 알고리즘을 기본 서명 방식으로 채택합니다.
- **향후 확장성(Future-proofing)**: 현재는 단일 인증 서버 아키텍처이므로 대칭키 방식(HS)을 사용하지만, 추후 MSA 환경으로 전환하여 인증 서버(Auth)와 리소스 서버(API)가 물리적으로 분리될 경우를 대비해 `RS256` (비대칭키)으로 쉽게 교체할 수 있도록 `JwtTokenProvider`의 인터페이스를 추상화 수준에서 설계합니다.

## 🔍 결정 근거 (Rationale)

### 1. 보안 강도 및 충돌 저항성 (Security & Collision Resistance)
- SHA-512는 SHA-256에 비해 해시 공간이 기하급수적으로 크기 때문에 무차별 대입 공격(Brute-force) 및 해시 충돌(Collision) 저항성이 월등히 높습니다.
- 양자 컴퓨팅의 발전 속도를 고려할 때, 512비트 해시는 시스템의 보안 수명을 비약적으로 연장시킵니다.

### 2. 하드웨어 최적화 (Hardware Optimization)
- 흔한 오해와 달리, 현대의 64비트 아키텍처(대부분의 클라우드 서버 환경)에서는 내부 레지스터 처리 구조로 인해 **SHA-512가 SHA-256보다 처리 속도가 더 빠르거나 동등한 성능**을 냅니다.
- 즉, 성능적 손실 없이 보안 강도를 2배 이상 끌어올릴 수 있는 '공짜 점심(Free Lunch)'에 가깝습니다.

## ⚖️ 대안 평가 (Alternatives)

| 알고리즘 | 장점 | 단점 | 탈락/보류 사유 |
|:---|:---|:---|:---|
| **HS256** | - 표준적으로 가장 널리 쓰임<br>- 토큰 페이로드가 미세하게 작음 | - 상대적으로 낮은 보안 강도 | 엔터프라이즈 보안 요건(강력한 방어)에 미달 |
| **RS256** | - 비대칭키 방식이라 개인키 노출 위험이 극히 낮음<br>- MSA 환경의 분산 인가에 최적화 | - 서명 연산 비용(CPU)이 높음<br>- 인증/인가가 같은 서버에서 일어나는 현재 구조에선 불필요한 비용 | **현재는 보류**. 단, 인프라가 MSA로 확장될 경우 즉각 전환 예정 (코드 추상화 완료) |

## 🚀 결과 및 영향 (Consequences)
1. **제약 사항**: HS512 알고리즘의 강력함을 유지하기 위해, `JWT_SECRET` 환경 변수는 반드시 **최소 64바이트(512비트)** 이상의 무작위 문자열이어야만 애플리케이션이 기동되도록 검증 로직을 추가했습니다.
2. **보안성**: 토큰 위변조에 대한 저항력이 금융권 요구사항 수준으로 격상되었습니다.
